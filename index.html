프로젝트 개요

`NEXT DEBUT`는 연습생 육성 게임, 커뮤니티, 상점, 팬미팅, 결제 기능이 하나의 서비스 흐름으로 이어지는 웹 프로젝트입니다. 여러 도메인이 섞이는 구조에서 기능을 붙이는 것보다 흐름이 끊기지 않도록 계층과 책임을 나누는 것이 더 중요하다고 판단했습니다.

담당 역할

백엔드 중심으로 인증, 결제, 실시간 채팅, 커뮤니티 기능 흐름을 구현하고 `controller - service - repository - entity - dto` 구조를 기준으로 책임을 분리했습니다.

핵심 기여

로그인부터 결제, 채팅, 커뮤니티까지 사용자가 실제로 지나가는 흐름을 하나의 서비스 경험으로 이어지게 만든 점이 가장 큰 기여였습니다.

Problem

채팅 기능은 단순히 메시지를 보여주는 것보다, 여러 사용자가 동시에 접속하는 상황에서도 메시지가 안정적으로 전달되고 흐름이 자연스럽게 이어지는 것이 더 중요했습니다. 실시간 통신 구조가 불안정하면 메시지 지연이나 누락처럼 보이는 문제가 발생할 수 있어, 서버 기준의 일관된 메시지 처리 흐름이 필요했습니다.

Solution

WebSocket 기반으로 사용자 간 메시지가 즉시 오갈 수 있도록 실시간 채팅 흐름을 구성했습니다. 서버에서는 메시지 수신, 처리, 전달 순서를 명확히 분리해 채팅방 단위로 흐름이 섞이지 않도록 정리했고, 연결 상태와 예외 상황도 함께 고려해 실시간 응답성이 유지되도록 구현했습니다.

Result

메시지 전달 흐름이 더 안정적으로 유지되면서 사용자가 실시간 대화를 자연스럽게 이어갈 수 있는 채팅 경험을 만들었습니다. 기능 나열보다 서버 측 메시지 흐름 제어가 품질을 좌우한다는 점을 프로젝트 안에서 직접 보여줄 수 있게 되었고, 포트폴리오에서도 백엔드 중심 문제 해결 사례로 설명하기 쉬워졌습니다.

Tech Decision

WebSocket은 실시간 양방향 통신이 필요한 채팅 기능에 가장 적합한 방식이어서 선택했습니다. 요청-응답 기반 구조보다 서버와 클라이언트가 연결을 유지한 채 즉시 메시지를 주고받을 수 있어, 채팅방 단위의 메시지 흐름을 더 안정적으로 관리하기에 유리했습니다.

홈 커뮤니티 육아정보 동네마켓 채팅 마이페이지 관리자
  • 홈: 서비스 소개, 주요 커뮤니티 진입, 캘린더와 추천 콘텐츠 배치
  • 커뮤니티·육아정보: 게시판 탐색, 카테고리 분류, 반응과 댓글 흐름 구성
  • 동네마켓·채팅: 위치 기반 거래 정보와 실시간 소통 기능 연결
  • 마이페이지·관리자: 회원 정보, 활동 이력, 운영 데이터 관리 구조

DATA MODEL

ERD 구조도

User Post Comment Category Market ChatRoom Reaction Report
  • 사용자(User)를 중심으로 게시글, 댓글, 반응, 신고 데이터가 연결되는 구조
  • 카테고리(Category) 분리를 통해 육아정보와 커뮤니티 게시판 탐색 흐름을 정리
  • 동네마켓(Market)과 채팅방(ChatRoom)을 연동해 지역 기반 소통 기능을 확장 가능하게 설계
  • 운영 관점에서는 신고(Report)와 게시글 데이터를 기준으로 관리자 모니터링이 가능하도록 구성

프로젝트 개요

`Moms Log`는 육아 정보 탐색, 공감형 커뮤니티, 지역 기반 소통을 하나의 플랫폼 안에서 연결한 서비스입니다. 정보성 콘텐츠와 커뮤니티 기능이 동시에 커지는 구조라서, 사용자가 원하는 정보에 빠르게 도달할 수 있는 데이터 구조와 카테고리 설계가 중요했습니다.

담당 역할

Django 기반 커뮤니티 기능과 정보형 콘텐츠 구조를 중심으로 게시판, 반응, 위치 기반 요소, 실시간 소통 기능이 확장 가능한 형태로 연결되도록 구현했습니다.

핵심 기여

정보 탐색과 커뮤니티 이용 흐름이 따로 놀지 않도록 게시판, 카테고리, 위치 정보 구조를 하나의 사용자 경험으로 정리한 점이 핵심 기여였습니다.

Problem

채팅 기능은 사용자가 바로 반응을 주고받을 수 있어야 했지만, 실시간 통신에서는 메시지 전달이 늦어지거나 흐름이 끊겨 보이면 사용 경험이 크게 떨어질 수 있었습니다. 특히 커뮤니티 서비스 안에 채팅 기능이 함께 들어가는 구조라서, 단순 연결보다 안정적인 메시지 흐름 관리가 중요했습니다.

Solution

실시간 채팅 흐름이 자연스럽게 이어지도록 서버 측에서 메시지 처리 순서와 채널 구조를 명확히 나누어 구성했습니다. 사용자가 보낸 메시지가 즉시 반영되도록 양방향 통신 구조를 기준으로 설계하고, 채팅방 단위의 흐름이 섞이지 않도록 처리 로직을 분리해 실시간 응답성을 높였습니다.

Result

채팅 기능이 단순 부가 기능이 아니라 서비스 안에서 자연스럽게 이어지는 실시간 소통 경험으로 정리되었습니다. 메시지 전달 안정성이 높아지면서 흐름이 더 매끄럽게 보이게 되었고, 포트폴리오에서도 백엔드 중심의 실시간 통신 문제 해결 사례로 설명할 수 있게 되었습니다.

Tech Decision

실시간 채팅에는 양방향 통신 구조가 필수적이기 때문에 WebSocket 기반 접근이 적합했습니다. 커뮤니티 서비스 특성상 메시지가 즉시 반영되어야 사용 흐름이 자연스럽게 이어지므로, 서버가 연결 상태를 유지하며 메시지를 처리하는 구조가 요청-응답 방식보다 더 잘 맞았습니다.

WHAT I LEARNED

프로젝트를 하며
중요하게 느낀 것

프로젝트를 진행하며 단순한 기능 구현보다 예외 상황을 어떻게 처리하고, 설정을 어떻게 분리하며, 협업 과정에서 충돌을 줄일 수 있는 구조를 설계하는 것이 서비스의 안정성과 유지보수성을 결정짓는 핵심이라는 점을 체감했습니다.

특히 기능이 늘어날수록 코드와 설정이 뒤섞이면서 관리와 확장이 어려워지는 문제를 경험했고, 이를 통해 역할과 책임을 분리하는 구조의 중요성을 이해하게 되었습니다.

이 경험을 통해 이후 개발에서는 기능 구현 이전에 구조와 흐름을 먼저 정리하고, 유지보수를 고려한 설계를 기준으로 삼게 되었습니다.

또한 포트폴리오를 정리하며 기능을 나열하는 것보다 문제를 어떻게 정의하고 해결했는지, 그리고 그 과정에서 어떤 판단을 했는지를 드러내는 것이 개발자의 역량을 더 잘 전달한다는 것을 깨달았습니다.
